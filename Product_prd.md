# Product Requirements Document — Local Wikipedia RAG

**Project:** BLG483E – Project 3
**Status:** Implemented
**Audience:** Developers and AI coding assistants who need to rebuild,
extend, or evaluate this system.

---

## 1. Problem statement

Build a ChatGPT-style question-answering system that can discuss famous
people and famous places, using only resources that run on a local
laptop. No external LLM, embedding, or search APIs are permitted.

The system must demonstrate the standard Retrieval-Augmented Generation
loop end-to-end: ingest documents, chunk them, embed locally, store in
a vector database, retrieve relevant context for each user query, and
generate a grounded answer with a local language model.

---

## 2. Users and use cases

The system has one user type: a person asking factual questions about
famous people or places via a chat interface.

**Primary use cases**

1. *"Who was X?"* — biographical or descriptive questions about a
   single entity.
2. *"What did X do?" / "Where is X located?"* — specific factual
   lookups.
3. *"Compare X and Y."* — side-by-side comparisons of two entities,
   possibly of different types (e.g. one person and one place).
4. *Catch-all topical queries* — *"Which famous place is in Turkey?"*,
   *"Which person is associated with electricity?"* — where the user
   does not name the target entity.
5. *Out-of-corpus questions* — the system must recognise these and
   refuse rather than hallucinate.

---

## 3. Functional requirements

### 3.1 Ingestion

- **FR-1.** The system shall ingest at least 20 famous people and 20
  famous places from English Wikipedia.
- **FR-2.** The required minimum entity set from the brief must be
  included.
- **FR-3.** Articles shall be saved as plain text under `data/`,
  organised by type (`data/person/`, `data/place/`).
- **FR-4.** Re-running ingestion shall skip already-fetched articles
  by default and provide an explicit `--force` flag for full re-fetch.

### 3.2 Chunking

- **FR-5.** Articles shall be split into overlapping chunks suitable
  for embedding.
- **FR-6.** Chunk size and overlap shall be configurable in one place
  (`src/config.py`).
- **FR-7.** The chunker shall prefer paragraph or sentence boundaries
  over arbitrary character cuts where they exist within a small
  look-back window.

### 3.3 Embedding and storage

- **FR-8.** A local embedding model (Ollama or sentence-transformers)
  shall produce one vector per chunk. No external API may be called.
- **FR-9.** Embeddings shall be stored in a persistent vector database
  on disk so reruns of the app do not require re-embedding.
- **FR-10.** Each stored chunk shall carry metadata: `type`, `title`,
  and `source_file`.
- **FR-11.** A single collection with metadata filtering (Option B in
  the brief) shall be used.

### 3.4 Retrieval

- **FR-12.** Given a user query, the system shall classify it as
  `person`, `place`, or `both`.
- **FR-13.** When the classification is `person` or `place`, retrieval
  shall apply the corresponding metadata filter. When `both`, retrieval
  shall search the full collection.
- **FR-14.** Retrieval shall return the top-K chunks (K configurable)
  with similarity scores and metadata.
- **FR-15.** Routing logic may be keyword-based or rule-based.

### 3.5 Generation

- **FR-16.** Answers shall be generated by a local language model via
  Ollama.
- **FR-17.** The prompt shall instruct the model to use only the
  retrieved context.
- **FR-18.** When the corpus does not contain the answer, the system
  shall reply *"I don't know based on the provided information."*
  This contract shall be enforced both via prompt instruction and
  via a programmatic check on retrieval scores or empty model output.
- **FR-19.** Generation shall support streaming output (token-by-token).

### 3.6 Chat interface

- **FR-20.** A Streamlit web UI shall provide:
  - a chat input,
  - streamed assistant responses,
  - a panel showing the source chunks used for each answer,
  - a "clear chat" / reset control,
  - the routing decision and per-query latency for retrieval and
    generation.
- **FR-21.** The user shall be able to toggle the source panel
  on or off.

### 3.7 Build automation

- **FR-22.** A single command (`python build_index.py`) shall perform
  ingestion and indexing end-to-end.
- **FR-23.** The repository shall include a `requirements.txt` with
  the project's top-level dependencies.

---

## 4. Non-functional requirements

- **NFR-1.** The full system shall run on a laptop with 8 GB RAM.
  16 GB is the comfortable target.
- **NFR-2.** Setup time, excluding model downloads, shall be under
  20 minutes.
- **NFR-3.** Per-query end-to-end latency on CPU shall be under 15
  seconds for typical questions.
- **NFR-4.** No data shall leave the user's machine. Wikipedia
  fetches at index time are the only outbound network calls.
- **NFR-5.** The repository shall be runnable by a third party
  following only the README, with no additional guidance.

---

## 5. Out of scope (v1)

- Conversational memory across turns ("she" / "that one").
- User authentication or multi-user state.
- Full-text keyword search alongside vector search (hybrid retrieval).
- Reranking with a cross-encoder.
- GPU acceleration.
- Languages other than English.
- Wikipedia content beyond the configured entity list.

---

## 6. Acceptance criteria

The system is considered complete when all of the following pass:

| # | Test | Expected behaviour |
|---|------|--------------------|
| AC-1 | "Who was Albert Einstein and what is he known for?" | Factual answer mentioning relativity / E=mc². |
| AC-2 | "Where is Mount Everest?" | Answer mentions the Himalayas / Nepal-Tibet border. |
| AC-3 | "Compare Lionel Messi and Cristiano Ronaldo." | Structured comparison with both names appearing. |
| AC-4 | "Which famous place is located in Turkey?" | Names Hagia Sophia and/or Topkapı Palace. |
| AC-5 | "Who is the president of Mars?" | Returns the canonical *"I don't know..."* refusal. |
| AC-6 | "Tell me about a random unknown person John Doe." | Returns the canonical refusal. |
| AC-7 | Streamlit UI loads at <http://localhost:8501> | Sidebar, chat input, source panel, latency display visible. |
| AC-8 | Fresh clone + README walkthrough | Instructor reaches a working app with no extra guidance. |

---

## 7. Open questions / future work

- **Hybrid retrieval.** Adding a BM25 keyword channel alongside vector
  search would improve named-entity queries further.
- **Threshold tuning.** The programmatic-refusal threshold is currently
  conservative; an evaluation set could tune it per-corpus.
- **Larger LLM.** Trying `llama3.1:8b` would improve answer quality at
  the cost of ~3x latency on CPU.
- **Citations.** Inline `[1]`-style citations referencing the source
  panel would strengthen verifiability.
